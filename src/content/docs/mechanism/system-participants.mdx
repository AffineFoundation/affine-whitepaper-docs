Affine operates as a decentralized evaluation network built on Bittensor (Subnet 120). The system comprises three principal roles, each with distinct responsibilities and trust assumptions. Table 1 at the end of this section summarizes all three at a glance.

## Miners

Miners are independent participants who train, host, and submit machine learning models. A miner's workflow proceeds through four steps:

1. Obtain a base model from the network via `af pull`
2. Improve the model through reinforcement learning or other training methods
3. Deploy the model as a serverless inference endpoint on Chutes
4. Commit the deployment metadata — including the Hugging Face repository, revision hash, and Chute identifier — to the Bittensor blockchain

Each hotkey may hold exactly one active commitment at any time. To ensure fair comparison, all submitted models must conform to the Qwen3-32B architecture (hidden_size=5120, num_hidden_layers=64), and quantized models are rejected.

The entire workflow is driven by the `af` CLI. For full setup instructions and a step-by-step walkthrough, see the [Complete Miner Guide](https://github.com/AffineFoundation/affine-cortex/blob/main/docs/MINER.md).

## Validators

Validators are blockchain-registered nodes responsible for setting on-chain weights. Validators fetch pre-computed normalized weights from the backend API and submit them to the Bittensor chain, preserving the decentralized weight-setting protocol.

For setup, configuration, Docker deployment, performance monitoring, and on-chain weight management, see the [Complete Validator Guide](https://github.com/AffineFoundation/affine-cortex/blob/main/docs/VALIDATOR.md).

> **Note:** Task orchestration — dispatching tasks to miner endpoints, collecting results, and submitting scored samples — is handled by backend infrastructure rather than a separate on-chain role.

## Environments

Environments are the evaluation substrates that make the entire pipeline meaningful — they are the containerized services deployed via Affinetes that present tasks to models and return scores. Affine defines a library of canonical environments (currently eighteen), spanning:

- Deduction and abduction
- Code generation and code editing
- Logic puzzles
- Game-theoretic problems
- Browser-based web interaction
- Navigation, memory, and knowledge evaluation

A dynamically configured subset of these environments is enabled for scoring at any given time. Each environment exposes a standard interface: either a single-turn `evaluate` call or a multi-turn OpenEnv protocol (`reset` -> `step` -> `stop`). The active environment set is retrieved at scoring time from a centralized system configuration endpoint, allowing the suite to expand without changes to the scoring pipeline.

---

The following table summarizes the three roles, their functions, trust models, and key constraints.

**Table 1. System Participants and Responsibilities**

<Table>

| Role | Primary Function | Trust Model | Key Constraint |
|---|---|---|---|
| **Miner** | Train and deploy models; compete for emissions | Untrusted — incentive-aligned via scoring | Must conform to Qwen3-32B architecture; one active commitment per hotkey |
| **Validator** | Set on-chain weights on Bittensor chain | Blockchain-registered; fetches pre-computed weights | Does not perform scoring directly |
| **Environment** | Present tasks to models and return scores | Containerized via Affinetes; standardized interface | Exposes single-turn `evaluate` or multi-turn OpenEnv protocol |

</Table>


> **Legend:** "Untrusted" means the participant is assumed to act in self-interest; the system aligns incentives through scoring and anti-plagiarism mechanisms rather than requiring trust.
