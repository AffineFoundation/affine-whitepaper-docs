Scoring turns raw per-task results into on-chain emission. Affine uses a winner-takes-all Champion Challenge design: a single reigning champion holds 100% of the emission weight, and challengers must first survive a pairwise anti-copy filter and then dominate the champion at a qualifying checkpoint before they can take the crown. Anti-exploit defense and on-chain settlement are not separate mechanisms; they are subsections of this same pipeline.

### Stage 1 — Data Collection

The scorer rolls every `SampleSubmission` from the prior round into per-miner, per-environment averages and normalizes environment-specific score ranges to [0, 1]. The output is a clean matrix of per-miner per-environment scores with per-task vectors retained for Stage 2.

### Stage 2 — Champion Challenge

Six phases run in order on the matrix from Stage 1.

*Champion resolution.* The pipeline first identifies the current champion from persisted on-chain state (hotkey + revision). When the champion is temporarily unreachable, the crown is preserved — only a formal dethrone can transfer it. On first deployment, the system seeds the champion from the highest-rated legacy miner.

*Pairwise Pareto filter.* For every non-champion miner pair that shares at least `PARETO_MIN_WINDOWS × window_size` common tasks (default `PARETO_MIN_WINDOWS = 3`), the filter runs a strict Pareto comparison with a fixed margin of `PARETO_MARGIN = 4%`. Strict here means the miner must be dominated in *every* environment to be eliminated, which avoids false positives on legitimate fine-tunes that win some environments and lose others. Registration time (`first_block`) settles incumbency: the later-registered miner is eliminated when the earlier one dominates. Conversely, if a later-registered miner dominates an earlier one, the earlier one is terminated instead — real progress displaces stale models regardless of who came first. Termination is permanent; because a hotkey can only hold one active revision, there is no recovery path.

*Checkpoint-gated comparisons.* Challenger-vs-champion comparisons are gated by the pool of common tasks. Each checkpoint requires `CP × window_size` common tasks, so `CP` advances whenever the shared task count crosses a new window boundary. A single comparison per round per challenger uses all available data up to that point, ensuring each check runs on a meaningfully different dataset rather than re-scoring on nearly the same sample.

*Dominance check.* At each post-warmup checkpoint, the challenger must exceed the champion in at least `WIN_MIN_DOMINANT_ENVS = 3` environments by the active `WIN_MARGIN`, and must not lose in any other environment. `WIN_MARGIN` scales linearly from `WIN_MARGIN_START = 2%` at the first post-warmup checkpoint to `WIN_MARGIN_END = 3%` at the dethrone checkpoint, so early comparisons are lenient and the decisive one is strict.

*Dethrone.* A challenger qualifies to take the crown when it has reached `CHAMPION_DETHRONE_MIN_CHECKPOINT = 10` and its most recent comparison was a win. If several challengers qualify simultaneously, the earliest registered takes precedence; geometric mean across environments (smoothed with `GEOMETRIC_MEAN_EPSILON = 0.1`) is used as a secondary tiebreaker. The first `CHAMPION_WARMUP_CHECKPOINTS = 2` comparisons are logged but do not count toward wins or losses — early checkpoints have too little data to be decisive, and the warmup protects strong challengers from noise-driven termination.

*Termination.* A challenger is permanently terminated when it accumulates `CHAMPION_TERMINATION_TOTAL_LOSSES = 3` total post-warmup losses or `CHAMPION_TERMINATION_CONSECUTIVE_LOSSES = 2` consecutive post-warmup losses. A loss at the dethrone checkpoint (`CP ≥ 10`) terminates immediately, since by that point the challenger has accumulated enough data that a decisive loss is not attributable to noise.

### Stage 3 — Weight Assignment

The reigning champion is assigned weight 1.0 and every other miner is assigned 0.0. If the champion is offline in the current round, its stored UID still receives weight 1.0 — the crown persists through absence.

### Anti-Exploit

The pairwise Pareto filter in Stage 2 is the primary anti-copy mechanism: because a copy matches the original on shared tasks, the copy cannot clear `PARETO_MARGIN` in every environment, and strict Pareto comparison terminates it before it ever challenges the champion. Registration order settles ties, so the earlier-committed original survives unless the later challenger is genuinely stronger. A background dual-signal anti-copy detector complements this on a 24-hour cycle by comparing model internals — hidden-state cosine similarity with a norm-deviation override, plus top-k logprob similarity — for any pair of miners with at least 30 shared tasks; both signals must agree before a flag is raised, which keeps the false-positive rate low and leaves legitimate fine-tunes untouched. Under the Champion Challenge structure, the defender advantage is the final layer: dethroning requires a genuine margin at CP ≥ 10, and a copy cannot clear that margin by reproducing the champion's scores.

### Settlement

The single-recipient weight vector from Stage 3 is picked up by validators and submitted to the Bittensor chain. Yuma Consensus aggregates stake-weighted medians across the validator set, and emissions flow to the champion hotkey. The single-recipient form simplifies consensus: whenever validators agree on who the champion is, the aggregate is identical to every individual submission.

$$
\mathrm\{Reward\}_\{\mathrm\{champion\}\} = 1, \qquad \mathrm\{Reward\}_\{i \neq \mathrm\{champion\}\} = 0
$$

A safety valve is retained. During inference-layer vulnerabilities, scoring bugs, or any other incident where continuing to pay the champion would reward the wrong party, the operations team can redirect champion weight to UID 0 (burn). This pauses emission without dismantling the champion state, so recovery only requires lifting the redirect once the issue is resolved.

**Table 3.2.** Champion Challenge parameters.

<Table>

| Parameter | Value | Role |
|---|---|---|
| `PARETO_MARGIN` | 4% | Fixed margin for pairwise Pareto and champion dominance |
| `PARETO_MIN_DOMINANT_ENVS` | 0 | Strict Pareto — must dominate in all environments |
| `PARETO_MIN_WINDOWS` | 3 | Windows of common tasks before pairwise filter fires |
| `WIN_MARGIN_START` | 2% | Margin at first post-warmup checkpoint (lenient) |
| `WIN_MARGIN_END` | 3% | Margin at dethrone checkpoint (strict) |
| `WIN_MIN_DOMINANT_ENVS` | 3 | Minimum environments where challenger must exceed champion |
| `GEOMETRIC_MEAN_EPSILON` | 0.1 | Smoothing for geo-mean tiebreaker |
| `CHAMPION_WARMUP_CHECKPOINTS` | 2 | Early checkpoints logged but not counted |
| `CHAMPION_DETHRONE_MIN_CHECKPOINT` | 10 | Minimum CP to qualify for dethrone |
| `CHAMPION_TERMINATION_TOTAL_LOSSES` | 3 | Total losses before permanent termination |
| `CHAMPION_TERMINATION_CONSECUTIVE_LOSSES` | 2 | Consecutive losses before permanent termination |

</Table>
