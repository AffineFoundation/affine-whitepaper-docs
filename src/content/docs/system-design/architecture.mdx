---
title: "4.1 Architecture"
---

The system is built in three layers, each bounded by a stable interface and each free to scale independently. The separation reflects the fact that the three layers have genuinely different resource shapes and different rates of change; coupling them would force the slowest-moving piece to constrain the fastest.

### Coordination Layer (affine-cortex)

The core protocol: task pool, executor workers, scorer, anti-copy detector, and validator-weight submission. This implements the mechanism in [§3](/mechanism/participants/). It is written in Python, backed by DynamoDB for the task pool and Redis for worker coordination, and runs on operator-managed infrastructure today. The coordination layer is CPU-bound — its load comes from database queries, score aggregation, and HTTP dispatch, not model inference.

### Environment Layer (Affinetes)

The containerization framework that packages each environment as an isolated, long-lived service. Affinetes handles image build, HTTP server injection, deployment, networking, and cleanup, so an author writes only the business logic ([§4.2](/system-design/affinetes/)). Each environment exposes a standard HTTP interface (single-turn `evaluate` and a gym-compatible multi-turn `reset/step/stop`) that the coordination layer speaks regardless of what the environment does internally. Also CPU-bound, but scales on a different axis: number of concurrent environment instances.

### Inference Layer (Chutes, Today)

Miner models are served through Chutes as OpenAI-compatible HTTP endpoints ([§4.3](/system-design/inference-layer/)). The inference layer is GPU-bound — every byte of throughput is a token being generated — and scales on a third axis: number of miner replicas and per-replica GPU memory. The environment layer speaks to the inference layer only over HTTP; environments do not load weights, allocate GPUs, or manage batching. That clean boundary is what lets Chutes be swapped for other OpenAI-compatible backends without touching environment code.

### Why the Separation

If coordination handled inference, one slow miner's forward pass would block every other task. If environments ran on the same machines as coordination, Docker-in-Docker overhead would crowd out the task pool. If inference were tightly coupled to environment state, swapping providers would require rewriting every environment. The three-layer split gives each layer the scaling freedom it needs and pushes coupling to a narrow, versioned HTTP interface in both directions.
