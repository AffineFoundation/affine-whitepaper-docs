# 3.3 Evaluation Loop and Sampling Strategy

The previous section described how individual tasks flow through the pipeline. This section zooms out to the loop-level view: how tasks are selected, rotated, and validated to ensure fair and robust evaluation across miners.

Affine's evaluation loop is designed to resist two failure modes common in static benchmarks:

- **Task saturation** — where models memorize fixed test sets
- **Sampling bias** — where some miners receive systematically easier or harder tasks

Three mechanisms work together to address these failure modes.

## Sampling Rotation

The system maintains a sampling list per environment that rotates tasks over time. The rotation works as follows:

- New tasks are added at the end of the list
- Consumed tasks are removed from the front
- Discovery order is preserved throughout

> **Why this matters:** A safety check prevents any sampling window from consuming more than 80% of the available dataset, ensuring that tasks remain renewable and that no miner can overfit to a narrow task distribution.

## Completeness Validation

Before a miner's scores enter the scoring pipeline, the system verifies that the miner has completed at least 90% of assigned tasks in each environment. Miners below this threshold are excluded from the current scoring round.

> **Why this matters:** This prevents partial evaluation from distorting rankings — a miner cannot cherry-pick easy tasks and skip hard ones to inflate their aggregate score.

## Common-Task Alignment

When comparing two miners, the scoring pipeline restricts comparison to the intersection of tasks that both miners have completed.

> **Why this matters:** This ensures that score differences reflect capability differences, not task-assignment differences. Without this constraint, a miner who happened to receive easier tasks could appear stronger than one who received harder tasks.
