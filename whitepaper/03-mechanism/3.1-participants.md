# 3.1 Participants

The system has two participant classes, miners and validators, both operating under the same publicly visible rules. Environments are covered separately in [§5](../05-environments/) — they are infrastructure rather than participants.

**Miners** are the permissionless participants. Any operator may `af pull` the shared Qwen3-32B base, run RL post-training against the environment suite, deploy the result on HuggingFace and Chutes behind an OpenAI-compatible endpoint, and commit the revision hash on-chain. A hotkey may hold only one active commitment at a time, and the base architecture is fixed — no quantization, no alternate backbones — so that scores across miners remain comparable and the anti-copy detector in [§3.2](3.2-scoring.md) is well-defined. See the [Miner complete guide](https://github.com/AffineFoundation/affine-cortex/blob/main/docs/MINER.md).

**Validators** are blockchain-registered nodes responsible for the full evaluation-to-settlement path. A validator runs the scheduler, executor, scorer, and anti-copy detector that drive the Champion Challenge pipeline in [§3.2](3.2-scoring.md), then submits the single-recipient weight vector on-chain. Scoring outputs are publicly observable and on-chain anchored, so any third party can independently reconstruct a round's settlement. See the [Validator complete guide](https://github.com/AffineFoundation/affine-cortex/blob/main/docs/VALIDATOR.md).

---

**Table 3.1.** Participant classes.

| Participant | Function | Trust model | Key constraint |
|---|---|---|---|
| **Miner** | Train, deploy, commit models | Permissionless; identity = hotkey + commit | One active commitment per hotkey; fixed base architecture |
| **Validator** | Run eval pipeline; submit on-chain weights | Stake-weighted registry; outputs publicly observable | Cryptographically signs every submission |
