# 5.1 Design Criteria

Environments are the substrate the rest of the system runs on. A weak environment quietly ruins everything above it — miners who game a bad scorer are indistinguishable on weights from those who solved a real problem, and the market stops pricing capability. Every environment admitted into the suite must satisfy five properties before it is eligible to contribute score.

### Real Task Structure

The task surface must come from a real problem domain, not a synthetic distribution chosen for scoring convenience. SWE-Infinite mines actual GitHub PRs; NavWorld models real-world city geography; LiveWeb Arena runs against live web pages. A miner that gets strong on a toy environment has learned the toy, not the underlying skill.

### Verifiable Scoring

Reward signals must be computable from observable outcomes, without a human in the loop at scoring time. Unit tests pass or fail; a plan reaches a destination or not; a game ends with a win, loss, or draw; a token distribution matches a reference or deviates from it. Anything requiring human judgment at scoring time would collapse throughput by orders of magnitude and introduce evaluator bias at the worst place in the pipeline.

### Anti-Contamination

The task distribution must be saturation-resistant by construction. Procedural generation (MemoryGym from seeds), rotating held-out splits (OpenSpiel trajectories), live external data (LiveWeb), or continuously-mined new tasks (SWE-Infinite PR ingest) all qualify; a fixed dataset does not. [§5.2](5.2-task-renewal.md) discusses renewal in depth.

### Bounded Per-Task Compute

Every environment enforces a hard wall-clock and token budget per task. This keeps one slow task from blocking a worker and makes coordination throughput predictable. An environment that can legitimately take 20 minutes for a single rollout must rearchitect into bounded segments or accept that only a small share of miners will be scored on it per round.

### Diverse Failure Modes

Each environment in the suite is admitted for a distinct capability axis — SWE, browser interaction, memory, tool-mediated planning, strategic reasoning, distributional alignment. Overlap is a cost: the Champion Challenge pipeline ([§3.2](../03-mechanism/3.2-scoring.md)) counts each environment independently, so two environments measuring the same underlying skill let a specialist earn dethrone credit twice. The bar is that a new environment must force a capability the suite does not already force.

**Table 5.1.** How the current six environments satisfy the five criteria.

| Environment | Real task surface | Verifiable scoring | Anti-contamination | Bounded compute | Distinct capability |
|---|---|---|---|---|---|
| **SWE-Infinite** | GitHub PRs | FAIL_TO_PASS + PASS_TO_PASS tests | Continuous PR ingest | Container + test timeout | Software engineering |
| **LiveWeb Arena** | Live web pages | Step rewards + terminal check | Live data + templates | Step + episode cap | Browser interaction |
| **MemoryGym** | Write-budgeted memory | Four-axis composite | Seeded procedural worlds | Write budget | Memory management |
| **NavWorld** | City graph + 6 MCP tools | 100-point rubric (code + LLM judge) | Weekly SHA256 salt rotation | Tool-call budget | Tool-mediated planning |
| **OpenSpiel** | 22 real games | Normalized outcome ∈ [0, 1] | Seed-indexed opponents | Move + time budget | Strategic reasoning |
| **DISTILL** | Top-miner teacher rollouts | Per-token KL to teacher | Self-expanding teacher pool | Rollout length cap | Distributional alignment |
