# 4.3 Inference Layer

Model serving is the third layer of the system and the one with the most asymmetric scaling shape. Every miner submission eventually becomes a long-running inference deployment answering token-generation requests from evaluation workers. The current system routes that traffic through Chutes; the interface between Affine and the inference provider is narrow enough that Chutes is replaceable without touching environment code.

### Model Deployment

Miners deploy their models through Chutes (Bittensor Subnet 64), which hosts each deployment as an OpenAI-compatible HTTP endpoint. When a miner commits on-chain, they point to a Chutes endpoint URL in the commitment metadata; the coordination layer reads that URL and forwards it to evaluation workers along with the task assignment. Chutes handles GPU allocation, model loading, and request batching — none of which Affine needs to reimplement. A miner can move from trained checkpoint to live scored endpoint in a single deployment step.

### The Clean HTTP Boundary

An environment never loads weights, allocates GPUs, or manages batching. When it needs the miner model to respond, it issues `openai.chat.completions.create` against the miner's endpoint and reads the response — text in, text out, with optional logprobs. Structurally, the environment treats the miner model the same way a user's application treats the OpenAI API. This narrowness is what makes the stack above the inference layer indifferent to how the model is served.

### Backend Substitutability

Because the contract is "OpenAI-compatible endpoint" and nothing else, the same environment code runs unchanged against alternative inference backends — Basilica, Targon, a self-hosted vLLM, or an operator-run cluster. Swapping backends means changing a base URL in the miner's commitment metadata; it does not touch any environment, scorer, or evaluation result.

### Forward Path

The roadmap item in [§6](../06-roadmap.md) around Affine-operated or hybrid inference infrastructure is a hedge against correlated failures of a single third-party provider, not a replacement of Chutes. The intent is to reduce the blast radius of any one inference outage and to give the operator direct levers over GPU availability during high-demand periods — while keeping the same HTTP contract so that nothing above the inference layer has to change.
